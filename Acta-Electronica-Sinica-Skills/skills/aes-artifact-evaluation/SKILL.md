# 《电子学报》研究制品与代码/数据可用性

需要先厘清现状：与部分国际会议 (如 ACM/USENIX 系) 不同，《电子学报》(Acta Electronica Sinica, AES)
作为中国电子学会 (Chinese Institute of Electronics) 主办的中文学报，**目前没有独立的 artifact
evaluation 徽章委员会或强制制品评审环节**（截至 2026-07-09 官网 ejournal.org.cn 与期刊主页未见公开
的制品评审章程，标 **待核实**）。因此本技能的目标不是"过徽章"，而是：在没有强制要求的前提下，
**主动、规范地开放并披露研究制品**，让外审专家更愿意相信你的结果，从而提高录用概率与论文影响力。

> 关键判断：制品开放在《电子学报》是**加分项而非门槛**。把它当作增强 electronics / signal-processing
> 类工作可信度的杠杆，而不是行政流程。Acta / Electronica / Sinica 的评审看重结果可信，制品是最硬的证据。

## 一、什么算"研究制品"

电子信息类论文的制品通常包括：

| 类别 | 典型内容 |
|---|---|
| 算法代码 | 信号处理/通信/机器学习算法的实现与脚本 |
| 数据集 | 采集的信号、图像、测量数据或仿真数据 |
| 仿真工程 | MATLAB/Simulink、ADS、HFSS、Vivado 等工程文件 |
| 硬件设计 | 电路原理图、PCB、FPGA/HDL 源码、约束文件 |
| 训练模型 | 网络权重、配置与评测脚本 |
| 复现说明 | README、环境依赖、运行步骤、预期输出 |

## 二、为何主动开放（对录用的实际帮助）

- 外审专家难以在数周内重跑复杂实验，一个**结构清晰、可一键运行**的仓库能显著降低其对结果真实性的疑虑。
- 开放制品使论文更易被后续工作引用、复现与扩展，提升长期影响力。
- 在修回阶段，指向具体代码/数据往往比长篇文字辩解更有说服力（见 `aes-author-response`）。

## 三、双盲阶段的匿名托管

由于本刊采用双盲评审，投稿阶段的制品链接**不得泄露作者身份**：

- [ ] 使用匿名仓库服务（如 anonymous 化的 Git 托管）或匿名压缩包作为附件上传。
- [ ] 仓库内 README、LICENSE、提交历史、作者字段中不出现真实姓名/单位/邮箱。
- [ ] 数据文件元信息、代码注释、路径名不泄露单位标识。
- [ ] 录用后再替换为正式的、署名的公开仓库地址（见 `aes-camera-ready`）。

## 四、制品打包最小规范

```
artifact/
├── README.md         # 一页说明：环境、依赖、运行、预期输出、数据来源
├── LICENSE           # 开源许可证（如 MIT / Apache-2.0）
├── env/              # requirements.txt / environment.yml / Dockerfile
├── src/              # 源码，入口脚本清晰
├── data/             # 数据或下载脚本（大数据给外链+校验和）
├── scripts/          # 一键复现主结果的脚本 run_all.*
└── results/          # 预期输出样例（图/表/日志）供比对
```

- README 必须让"没有背景的人"也能按步骤跑出论文主结果的核心图表。
- 大数据集用稳定外链 + 校验和 (MD5/SHA256)，不要把数 GB 数据塞进仓库。
- 固定随机种子、记录关键超参数，尽量给出运行时间与硬件规格。

## 五、正文与附录中的制品披露

- 在实验章节或附录用一小节写明：制品含哪些内容、托管在何处、如何运行、对应论文哪些图表。
- 对硬件/FPGA 工作，给出工具链版本（如 Vivado 版本）、器件型号、约束与时序报告位置。
- 对数据集工作，说明采集设备、采样率、规模、标注方式与授权/伦理合规性。

## 六、常见问题与规避

| 问题 | 规避 |
|---|---|
| 仓库泄露作者身份 | 双盲期用匿名托管，逐项清查元信息 |
| 依赖缺失跑不起来 | 提供 Dockerfile 或锁定版本的依赖清单 |
| 数据无法获取 | 给可下载外链 + 校验和 + 授权说明 |
| 结果不可复现 | 固定种子、记录硬件、提供预期输出比对 |
| 许可证缺失 | 明确开源许可证，说明第三方数据/代码授权 |

## 七、制品自检清单

- [ ] 制品类别已识别（代码/数据/仿真/硬件/模型）。
- [ ] 双盲期匿名托管，元信息无泄露。
- [ ] README 可让第三方独立跑出主结果。
- [ ] 依赖/环境可复现（锁版本或容器化）。
- [ ] 大数据用外链 + 校验和。
- [ ] 正文/附录已规范披露制品位置与对应关系。
- [ ] 录用后替换为署名公开仓库并补许可证。

## 八、按学科细分的制品要点

不同电子信息子方向的制品重点不同，投《电子学报》时应对症准备：

| 子方向 | 制品重点 |
|---|---|
| 信号与信息处理 | 原始/仿真信号、预处理脚本、指标(SNR/BER)计算代码、随机种子 |
| 通信工程 | 信道模型、蒙特卡洛仿真配置、误码统计脚本、参数表 |
| 集成电路/EDA | HDL 源码、约束文件、综合与时序报告、器件与工具版本 |
| 电磁场与微波 | 仿真工程(HFSS/CST)、结构参数、网格设置、实测对比数据 |
| 机器学习类 | 数据划分、模型权重、训练/评测脚本、超参数与算力记录 |

对每一类，README 都应让审稿人"照着就能重现论文主结果的核心图表"。硬件类若无法公开完整工程，
至少给出关键参数与足以复算的说明，并注明保留原因（如涉密或第三方授权，配合脱密要求）。

## 九、开放程度的分级

- **完全开放**：代码+数据+说明全公开，可复现性最强，最利于录用与后续引用。
- **部分开放**：核心代码公开、敏感数据以外链或申请方式提供，附充分说明。
- **说明为主**：因涉密/授权无法公开，则在正文给出足够复算的参数、流程与伪代码。

无论哪一级，都要**如实说明开放边界**，不夸大也不隐瞒，这本身就是《电子学报》审稿人看重的科研诚信。

## 输出格式（制品可用性报告）

```
【《电子学报》(AES) 研究制品可用性】
本刊制品评审现状：无强制徽章（待核实），主动开放为加分项
制品类别：<代码 / 数据 / FPGA 工程 / …>
双盲匿名：<通过 / 存在泄露点>
可复现性：<容器化 / 锁版本 / 需补>
数据获取：<外链+校验和 / 待补>
正文披露：<已写明位置与对应图表>
待核实：<本刊是否有制品相关政策>
```

即便《电子学报》(Acta Electronica Sinica, AES) 不强制制品评审，一份规范开放的制品也是让外审
专家信任你结果、让论文获得长期引用的最佳投资。
