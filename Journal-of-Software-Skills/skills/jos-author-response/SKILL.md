# 《软件学报》审稿意见答复与修回 (Journal of Software Author Response)

收到《软件学报》(Journal of Software, JOS) 的"修改后再审"意见后，修回说明（答复审稿人）的
质量往往决定下一轮是录用还是退稿。本刊大修常由原外审专家复审，敷衍会被识别。本技能给出
逐条回应的结构与分寸（流程背景见 [`jos-review-process`](../jos-review-process/SKILL.md)）。

## 一、修回总原则

- **逐条回应**：对每条意见单独回应，不遗漏、不合并含糊带过。
- **尊重且有据**：感谢审稿人，回应基于证据与论证，不情绪化、不辩护式对抗。
- **真正落实**：能改的就改，并在稿件中标出对应修改；不能改的说明理由。
- **前后一致**：多轮修回时，与上一轮答复、与稿件其他部分不矛盾。

## 二、意见分级策略

| 类型 | 策略 | 表达 |
| --- | --- | --- |
| 必须改（合理且可行） | 直接照做，标注修改位置 | "已按建议修改，见第 X 节/表 Y" |
| 可讨论（有分歧但合理） | 折中或补实验/补论证 | "我们理解该关切，补充了……以说明……" |
| 礼貌不改（误解或超范围） | 澄清+给依据，不硬顶 | "感谢指出，可能因表述不清，实为……已在第 X 节澄清" |

- 对"必须补的实验/数据"，尽量真的补；无法补时说明代价与替代验证。
- 不同审稿人意见冲突时，说明你的取舍依据，并让责任编委看到你的权衡。

## 三、修回说明结构

```text
尊敬的编辑、各位审稿专家：
  感谢对本文的评审意见。我们已逐条修改并说明如下（修改处在稿件中以颜色/下划线标出）。

【审稿人 1】
  意见 1：……
  回应：已修改。具体地，我们在第 3.2 节新增……（对应稿件第 X 页）。
  意见 2：……
  回应：我们补充了 RQ3 的跨项目实验（表 5），结果表明……
...
【审稿人 2】
  意见 1：……
  回应：……
【编委/主编意见】
  回应：……
```

## 四、稿件修改的对应标注

- 提交**修改稿 + 修回说明**两份；修改稿中用颜色/下划线标出改动，便于复审专家核对。
- 修回说明中每条回应指明**对应到稿件的具体位置**（节号/页/表号），不要只说"已修改"。
- 重大改动（新增实验、结构调整）单独列出，方便复审快速定位。

## 五、多轮修回的一致性

- 保留每一轮的意见与答复存档；新一轮回应不与旧轮矛盾。
- 若上一轮承诺的修改本轮才落实，明确说明；不要让审稿人重复提同一问题。
- 术语、数据、结论在全文与各轮答复间保持一致。

## 六、语气与分寸

- 全程使用尊重、专业的中文表达；对批评先致谢再回应。
- 不空洞附和（"完全同意"却不改），也不硬顶（"审稿人错了"）。
- 承认局限比掩饰更能取信；把未解决问题诚实写入威胁有效性或未来工作。

## 七、修回自检清单

```text
[ ] 每条意见都有单独回应，无遗漏
[ ] 每条回应指明稿件对应修改位置
[ ] 必须补的实验/数据已尽力补齐
[ ] 冲突意见的取舍有说明
[ ] 修改稿已标出改动，便于复审核对
[ ] 多轮之间答复一致，无自相矛盾
[ ] 语气尊重专业，既不附和也不对抗
[ ] 未解决问题诚实写入局限/未来工作
```

## 八、输出格式

```text
【修回就绪度】就绪 / 需补 / 重写
【意见清单】按审稿人整理，分级：必须改/可讨论/礼貌不改
【补实验计划】需补：________（可行性）
【对应标注】是否每条都指到稿件位置：________
【一致性】与前轮/全文是否冲突：________
【下一步】提交修改稿+修回说明，等待复审与主编终审
```

## 九、典型意见的应对范式

《软件学报》(Journal of Software) 外审常见几类意见，各有得体的应对范式，照搬空话最容易翻车：

- **"创新性不足/与已有工作区分不清"**：这是最需认真对待的一类。补强相关工作的差异定位
  （见 [`jos-related-work`](../jos-related-work/SKILL.md)），用一句话讲清增量，并在实验中给出
  区分性证据；切忌只在答复里辩解而不动正文。
- **"实验不充分/缺基线/缺统计检验"**：尽量补做实验（补基线、补数据集、加统计检验与效应量，
  见 [`jos-experiments`](../jos-experiments/SKILL.md)）；确实无法补的，说明代价并给替代验证。
- **"写作不规范/术语不一致/摘要空泛"**：逐项修订，统一术语，重写摘要，并在答复中指明改动
  位置；这类意见改起来直接，务必全部落实。
- **"某处方法/结论存疑"**：先判断是误解还是真问题。误解就澄清并改进表述；真问题就修正结论
  或缩小声称范围，诚实优于掩饰。
- **"英文摘要质量差"**：请专业人士润色 English Abstract，确保术语与正文一致、可独立阅读。

## 十、提交前最后一遍

```text
[ ] 修回说明 + 修改稿两份齐备，修改处已标注
[ ] 每条意见对应到稿件具体节/页/表
[ ] 承诺补的实验/数据已并入正稿
[ ] 术语、数字、结论全文一致
[ ] 与前几轮答复无矛盾
[ ] 语气尊重专业，无情绪化辩护
[ ] 英文摘要已润色（若被指出）
```

> 《软件学报》(Journal of Software) 编辑部与责任编委的具体通知为准。定稿见
