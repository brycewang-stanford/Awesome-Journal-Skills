# 《软件学报》附录与补充材料 (Journal of Software Supplementary)

本技能帮你决定《软件学报》(Journal of Software, JOS) 稿件里"什么放正文、什么放附录/补充
材料"。核心原则：**决定录用的内容必须在正文**，附录只承载支撑性细节。本刊补充材料的具体
接收方式以编辑部与模板为准（**待核实**，见
[`resources/official-source-map.md`](../../resources/official-source-map.md)）。

## 一、切分原则（按"决定性"划分）

- **必须在正文**：贡献陈述、核心方法、主要实验与结论、关键威胁有效性——任何影响审稿人接收
  判断的内容都不能移出正文。
- **可放附录/补充**：完整证明、额外实验、超参表、附加案例、数据统计明细、界面截图集等
  支撑细节。
- 判据：如果把某段移到附录会让审稿人无法判断贡献是否成立，它就必须留在正文。

## 二、补充材料类型

| 类型 | 内容 | 放置 |
| --- | --- | --- |
| 数学附录 | 完整推导/证明 | 附录 |
| 实验附录 | 额外 RQ、消融细节、完整结果表 | 附录/补充 |
| 数据附录 | 数据集统计、标注规范、样例 | 附录/补充材料 |
| 复现材料 | 代码、脚本、可复现包 | 见 jos-reproducibility |
| 多媒体附件 | 演示视频、交互式结果 | 在线附件（如支持） |

## 三、引用与可发现性

- 正文在恰当处引用附录/补充材料（"详见附录 A""完整结果见补充材料"）。
- 补充材料自带说明，能独立被理解，不依赖读者猜测。
- 多媒体/在线附件给出稳定链接与访问说明。

## 四、与可复现材料的关系

- 补充材料（叙述性支撑）与可复现材料（代码/数据/脚本）功能不同但相关。
- 复现包的组织与许可见 [`jos-reproducibility`](../jos-reproducibility/SKILL.md) 与
  [`jos-artifact-evaluation`](../jos-artifact-evaluation/SKILL.md)。
- 不要把决定性证据仅藏在复现包里而正文缺席——正文要能自洽支撑结论。

## 五、评审 vs 出版两阶段

- **评审阶段**：附录随稿件提交，帮助审稿人核查细节；若采用匿名，材料也需去除身份线索（本刊
  评审匿名要求以投稿指南为准，**待核实**）。
- **出版阶段**：定稿时确认附录/补充材料的最终形态与链接稳定性（见
  [`jos-camera-ready`](../jos-camera-ready/SKILL.md)）。

## 六、组织自检清单

```text
[ ] 决定录用的内容全部在正文，未被移到附录
[ ] 附录只放支撑性细节（证明/额外实验/数据明细）
[ ] 正文在恰当处引用了附录/补充材料
[ ] 补充材料能被独立理解，自带说明
[ ] 多媒体/在线附件有稳定链接与访问说明
[ ] 复现材料与叙述性附录分工清楚
[ ] 定稿时确认附录最终形态与链接
```

## 七、输出格式

```text
【切分就绪度】就绪 / 需调整
【正文自洽】是否有决定性内容被移出正文：________
【附录内容】清单：________
【链接稳定】多媒体/在线附件：________
【与复现材料关系】是否重复/缺位：________
【下一步】用 jos-reproducibility 组织复现包 / 用 jos-camera-ready 定稿
```

## 八、常见问题

- 把关键实验或证明塞进附录，正文无法自洽——审稿人会要求移回正文。
- 补充材料缺说明，审稿人看不懂。
- 多媒体附件链接不稳定或失效。
- 附录与正文数字不一致。

## 九、不同论文类型的附录侧重

《软件学报》(Journal of Software) 不同类型论文，附录/补充材料的侧重不同：

- **方法/技术类**：完整算法、复杂度分析、参数敏感性实验、失败案例分析可入附录；核心方法与
  主结果留正文。
- **系统类**：系统架构细节、配置、额外性能测量、部署说明可入附录；关键设计取舍与主评测
  留正文。
- **实证研究类**：完整数据统计、标注规范与手册、附加变量分析、问卷全文可入附录；主要发现
  与威胁有效性留正文。
- **综述类**：完整文献清单、分类明细表、检索记录可入补充材料；分类框架与开放问题留正文。
- **形式化/理论类**：完整证明与引理链可入附录；关键定理与证明思路留正文。

## 十、篇幅与取舍

- 附录不是正文超篇幅的"泄洪区"：正文本身要在栏目篇幅期望内自洽（见
  [`jos-submission`](../jos-submission/SKILL.md)）。
- 把"读者验证细节需要、但不影响接收判断"的内容下沉到附录，保持正文流畅。
- 补充材料过多会增加审稿负担，按"是否真的帮助理解或复现"取舍，宁精勿滥。

## 十一、快速对照

```text
[ ] 论文类型对应的附录侧重是否合理
[ ] 正文在篇幅内自洽，未靠附录撑结论
[ ] 附录内容"验证需要但不决定接收"
[ ] 补充材料精炼，不增加无谓审稿负担
```

归根结底，向《软件学报》(Journal of Software) 投稿时，正文与附录的分工要服务于同一个目标：
让审稿人在正文里就能完整判断贡献成立与否，再在附录里按需核验细节。正文自洽是底线，附录是
支撑；两者边界清楚、相互引用得当，稿件才既紧凑又经得起推敲。

## 十二、类型化输出格式

```text
【论文类型】方法/系统/实证/综述/理论
【附录侧重】按类型应放入附录的内容：________
【正文自洽】是否有决定性内容误入附录：________
【篇幅取舍】补充材料是否精炼：________
【下一步】用 jos-camera-ready 在定稿时确认附录最终形态与链接
```

> (Journal of Software) 官网投稿指南与 2021 版模板为准。
